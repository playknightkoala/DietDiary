## 1. 結論

不需要換 PostgreSQL、導入 Redis、SSE/WebSocket 或大幅重構。最有價值的效能工作只有三項，都不改變功能、可獨立上線、可客觀驗收：

1. **nginx 啟用 gzip ＋ Vite hashed assets 設 immutable 長期快取**。
2. **批次化 `getDayJson()`**：代表性每日頁面 36-37 次 SQL → 固定 6-8 次。
3. **前端直接使用 mutation 回傳的 `DayData`**，移除多餘的 day＋marks GET。

其他項目一律由量測結果決定，不直接全部實作。

## 2. 現況基線

- Backend：Node/Express/better-sqlite3（WAL）；本機 DB 約 104 KB，尚無大資料量壓力。
- Frontend：React 19/Vite/Zustand；production JS 約 366 KB（gzip 估約 105 KB）。
- 圖片前端先壓長邊 640px／q0.7；Zustand 細粒度 selector，無證據顯示重繪是瓶頸。

## 3. P0：nginx 傳輸與靜態快取

**問題**：`frontend/nginx.conf` 無 gzip 設定，366 KB JS 原樣傳輸。預設 ETag/Last-Modified 可條件式重驗，但 `/assets/*` 帶 content hash，可設 immutable 完全省掉重驗往返。

```nginx
gzip on;
gzip_vary on;
gzip_min_length 1024;
gzip_types text/css application/javascript application/json image/svg+xml;

location /assets/ {
    try_files $uri =404;
    add_header Cache-Control "public, max-age=31536000, immutable";
}
location = /index.html {
    add_header Cache-Control "no-cache";
}
```

**邊界**：`index.html` 與 `changelog.json` 不可 immutable；`/api` JSON 由同組設定壓縮（瀏覽器直連不需 `gzip_proxied`），部署後以 `Content-Encoding: gzip` 實測；JPEG 不再 gzip；`/uploads` 已由 Express 設 30 天 immutable（`index.ts:39`），維持現狀。

**驗收**：JS transfer 366 KB → ~105 KB；`/assets/*` 二訪無網路重驗；HTML 與 changelog 仍能拿到新版。

## 4. P1：批次化 `getDayJson()`（`backend/src/helpers.ts:486-526`）

**問題**：會員日記、營養師查看（`pro.ts`）與所有寫入 API 的共用熱路徑。SQL 次數隨當日記錄數線性成長：day row 1＋entries 1＋每筆 entry 2（ratings＋留言數）＋water 1＋每筆 log 1＋ex 1＋每筆 log 1＋summary 1＋feedback 1。10 飲食/8 喝水/3 運動 = **37 次**（無 summary 時 36），better-sqlite3 同步查詢佔用 event loop。

**批次方案**：
1. 當日 entries：1 次。
2. 當日 ratings：1 次固定 JOIN（已核對 `photo_ratings` 結構，`db.ts:84-91`）：
   ```sql
   SELECT pr.entry_id, pr.photo, pr.rating
   FROM photo_ratings pr
   JOIN entries e ON e.id = pr.entry_id
   WHERE e.user_id = ? AND e.date = ?;
   ```
3. 留言數：1 次 `WHERE user_id = ? AND target IN (...) GROUP BY target`——帶 `user_id` 即命中現有 `(user_id, target)` 索引，**不新增索引**。
4. summary＋feedback：1 次（LEFT JOIN 或同等單次查詢）。
5. Node 端依 `entry_id`／**帶前綴的 target 字串**（`entry:12`、`water:12` 是不同目標）分組組回原 JSON。

**空集合防護**：當日沒有任何 entry/water/ex target 時，不要組動態 `IN ()` SQL（SQLite 語法錯誤），直接回傳空的 count map：

```ts
const commentCounts = targets.length > 0 ? queryCounts(targets) : new Map();
```

順帶讓 `countComments`（`helpers.ts:291`）與 entry 刪除（`entries.ts:148`）帶入已知 owner `user_id` 用上索引。

**驗收**：固定測資（10/8/3）下 36-37 次 → 固定 6-8 次；回傳 JSON 與改造前**完全相容**（ratings、commentCount、AI summary/feedback 不可遺失）。

## 5. P1：mutation 回傳值寫入 Zustand（前端）

**問題**：喝水、運動、身體 API 已回傳完整 `DayData`（`days.ts:47,61,75` 等），但 `WaterModal.tsx:36-39`、`ExerciseModal.tsx:37-41` 只寫進 modal 區域 state，接著 `refresh()` 再打 day＋marks 兩個 GET——day GET 正是上節的 N+1 路徑。

**store 增加**：
- `replaceDay(date, day)`：僅 `date === selected` 時替換全域 day。
- `markDate(date, hasData)`：寫入後樂觀設 true；刪除後依回傳 DayData 以 `dayHasData()` 判斷。

飲食 PATCH 回單筆 Entry 由前端局部替換；**跨日移動**同時影響兩天的 marker，保留定點 refetch 即可，不為完全樂觀更新增加複雜狀態邏輯。

### `dayHasData` 契約（前後端必須一致）

`hasData` **不能**用 `day.entries.length > 0` 判斷——新增飲食時後端先建立空白 entry、之後才 PATCH 填內容（`entries.ts` 的「第一次從空白變成有內容＝發布新貼文」邏輯即此設計），空白 entry 不應讓月曆亮燈。前端判斷必須複製後端 `getMarkedDates` 的規則（`helpers.ts:537-554`）：

```ts
// 對齊 backend getMarkedDates（helpers.ts:543-545, 551）＋ entryHasData（helpers.ts:528）
export function dayHasData(day: DayData): boolean {
  const hasBody = Object.values(day.body).some((v) => v !== '');
  const hasEx = Number(day.ex.min) > 0 || day.ex.desc !== '';   // 用 days 快取欄位，和後端讀的同一份
  const hasEntry = day.entries.some(
    (e) => !!(e.desc || e.photos.length || Object.values(e.food).some((v) => v > 0))
  );
  return day.water > 0 || hasEx || hasBody || hasEntry;
}
```

注意 `hasEx`/`water` 建議直接用 `DayData` 的 `ex`/`water` 欄位——它們就是後端 `days` 表的快取（由 log 重算），與 `getMarkedDates` 讀的是同一份資料，比從 `exLogs`/`waterLogs` 重新聚合更不會歧義。`aiSummary` 不影響 marker。

**Marker 契約案例**（本專案沒有測試框架，以下作為手動驗收清單；若要自動化需先另行決定引入測試工具）：

| 情境 | marker |
|---|---|
| 只有空白 entry | false |
| entry 填入敘述／照片／任一份數 > 0 | true |
| 刪除最後一筆有效 entry | false |
| 只有喝水（water > 0） | true |
| 只有運動敘述、沒有分鐘 | true |
| 只有身體資料（任一欄非空） | true |
| 只有 AI summary | false |

**驗收**：喝水/運動/身體寫入後 Network 只有 mutation；主畫面與 modal 顯示一致；當天由無→有資料時 marker 立即出現，刪除最後一筆後 marker 正確消失。

## 6. P2：資料量成長後再處理（先量測）

**KB 全表向量掃描**（`kb.ts:112-121, 134-142`）：<1,000 筆先量測維持現狀；1,000-5,000 向量常駐記憶體、寫入時更新快取；>5,000 或 event-loop delay 過高導入 sqlite-vec/Qdrant 等 ANN，或移到 Worker Thread/embedding service。以 1k/5k/10k 假資料測 lookup p50/p95 與 event-loop delay，門檻只是測試起點。

**清單圖片**（`DayFeed.tsx:139`、`HistoryPickerSheet.tsx:79` 等）：先把 CSS `background-image` 改 `<img loading="lazy" decoding="async">`＋`object-fit: cover`；量測 transfer bytes 後仍高才另存 120-160px 縮圖（列表用縮圖、lightbox 用 640px）。未量測前不引入縮圖的儲存/清理/migration 複雜度。

**字型與 code splitting**：補 `fonts.gstatic.com` preconnect（`index.html:7` 目前只有 googleapis）；`display=swap` 已存在；依 Network 面板確認字重實際成本（CJK 有 unicode-range 分片），只刪真正沒用的；Admin/Dietitian 畫面可 `React.lazy()` 角色級拆分；vendor chunk 需另行評估 `manualChunks`，兩者是不同的最佳化。gzip 後整包 ~105 KB，非當前主瓶頸。

## 7. P3：以 query plan 與量測決定

**候選索引**（接近 production 的資料量上先 `EXPLAIN QUERY PLAN`，勿一次全建）：

```sql
CREATE INDEX idx_notifications_user_id ON notifications(user_id, id DESC);
CREATE INDEX idx_notifications_dedupe ON notifications(user_id, type, target, member_id, read);
CREATE INDEX idx_follows_member ON follows(member_id);
```

現有 `notifications(user_id, read)` 供 unread count，保留。

**AI feedback 查詢**（`helpers.ts:456`，每次 `/ai/comment`、`/ai/daily` 都執行）：優先改為兩個有界查詢（`vote = 1`/`vote = -1` 各 `ORDER BY created_at DESC LIMIT 3`）＋ `(user_id, vote, created_at DESC)` 索引；若改抓最近 50-100 筆由 Node 分組則用 `(user_id, created_at DESC)`。兩種索引不同時建。

**輪詢與日期快取**：背景分頁停止 60 秒通知/版本輪詢（`App.tsx:30-41`），回前景立即更新一次；日期 TTL cache 等 mutation 寫回/失效機制完成後再做。

**同步 I/O 與圖片處理**：先量 event-loop delay，再把 `writeFileSync`（`entries.ts:171`）、`copyFileSync`、`readFileSync`（`llm.ts:87`）、history 冷快取 SHA-1（`helpers.ts:132-144`）逐步改非同步；multer 2 MB × 10 是直連 backend 的理論上限，公開路徑受 nginx `client_max_body_size 5m` 限制；`shrinkJpeg()`（`llm.ts:61-79`）調整需同時測 CPU、檔案大小、品質與 OCR 準確度。

**Prepared statements**：better-sqlite3 無內建語句快取，熱 SQL 可提升為 module-level Statement，成本小於 N+1，最後做。

**SQLite `synchronous = NORMAL`**：durability 取捨非免費加速——確認 fsync 是瓶頸、可接受斷電遺失最近少量已提交寫入、備份/復原完善後，再比較 FULL/NORMAL 寫入 p95。目前寫入量小，不急。

## 8. 不建議項目

- 換資料庫／導入 Redis／SSE 或 WebSocket。
- 全面 React memoization（無重繪瓶頸證據）。
- 無 query plan 佐證就一次建立所有候選索引。
- 未量測就建立全量縮圖。
- **`requireRole`/`requireAI` 加 TTL 權限快取**：每次重讀 DB 是刻意設計（停權/降級即時生效，CLAUDE.md 明文記載）。
- **改動 `changelog.json` 的雙軌機制**：打包版同時是頁尾資料來源與即時抓取失敗的 fallback（`changelog.ts:33-35`），刻意設計，9.5 KB 收益不值得。

## 9. 量測方案

**Production 持續指標**：API route/status/duration p50/p95/p99；Node event-loop delay；前端 LCP/INP/JS transfer；每頁圖片請求數與 transfer bytes；KB 筆數與 lookup p95。

**Benchmark/診斷模式指標**：每請求 SQL 次數與累計耗時；history 冷快取 SHA-1 耗時；KB cosine CPU 與 1k/5k/10k benchmark；圖片上傳/sharp 的 event-loop delay 與 libuv worker pool 等待時間。

SQL 逐筆追蹤與完整 query logging **只在診斷模式啟用**，避免 production 全量 logging 自身增加延遲、I/O 與敏感資料暴露。

## 10. 驗收標準

1. nginx：JS transfer ~366 KB → ~105 KB（105 KB 是 Vite 估算，**以部署後實測為準**）；`/assets/*` 二訪不向伺服器重驗。實測方式：

   ```bash
   curl --compressed -I https://<host>/assets/<hash>.js
   # 確認 Content-Encoding: gzip
   # 確認 Cache-Control: public, max-age=31536000, immutable
   ```
   並用瀏覽器 Network 面板確認二次載入 hashed asset 無重驗請求。
2. day API：10/8/3 測資下 36-37 次 SQL → 固定 6-8 次；JSON 完全相容。
3. mutation：寫入後只有 mutation，無緊接的 day＋marks GET；主畫面、modal 與 marker 一致；marker 行為通過 §5 的契約案例清單。
4. KB：1k/5k/10k 測試同時記錄 lookup latency 與 event-loop delay。

## 11. 建議執行順序

1. 建立最小量測基線。
2. nginx gzip／immutable assets（P0）。
3. `getDayJson()` 批次化，留言查詢帶 `user_id`（P1）。
4. mutation 回傳值寫入 Zustand，移除重複 refetch（P1）。
5. query plan 確認 notifications/follows/AI feedback 索引，並限制 feedback 查詢。
6. 清單照片 lazy loading；背景分頁停輪詢。
7. 依量測處理同步 I/O、sharp、multer。
8. 依 KB benchmark 決定向量 cache/ANN。
9. 最後：code splitting、字重精簡、prepared statements、SQLite durability 取捨。
